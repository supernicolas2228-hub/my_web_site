# Два сервера (Москва + Европа) и домен на NetAngels

Ниже — два режима: **сейчас у вас один VPS** (Aeza) и **позже — два зеркала** (РФ + EU). Один домен (`truewebwork.ru`), один код; DNS настраиваете в панели регистратора (часто NetAngels).

## Как должно работать для людей (РФ без VPN, VPN, Google, Safari)

Ожидание: человек из **России без VPN**, человек **с VPN** (другой IP), переход **из Google** или **по прямой ссылке** в **Chrome / Safari** — везде **один и тот же сайт** (те же страницы и поведение).

Как этого добиться **пока сервер только один** (европейский Aeza):

1. В DNS (**NetAngels** или где висит зона) в **каждом** регионе GeoDNS — **Россия, СНГ, «мир», «по умолчанию»** — для **`@` и `www`** стоят **одни и те же A‑записи** на **публичный IPv4 этого VPS**. Тогда и резолвер в РФ, и резолвер «как у VPN», и робот Google попадают на **одну** машину с **одной** сборкой.
2. Канонический адрес в коде — **`https://www.truewebwork.ru`**; имя без `www` редиректит на `www`, если запрос дошёл до сервера.
3. **Яндекс** вы планируете не трогать, пока нет сервера в РФ — ок: в `robots.txt` он не закрыт, но **выдача Яндекса** и **гео‑DNS под РФ** — отдельный шаг, когда появится зеркало в России и вы захотите отдельную схему.

Если в GeoDNS для **России** указан **другой IP**, чем для «остального мира», а второго сервера с актуальным деплоем нет — у части пользователей сайт будет **не таким**, как у тех, кто попал на Aeza. Поэтому при **одном** VPS все регионы = **один IP**.

## Автоматизация NetAngels (почти «сделай за меня»)

Я не могу зайти в ваш аккаунт, но в репозитории есть скрипт: он сам получает токен API и **обновляет все A** для **`домен`** и **`www.домен`** на один IPv4, при необходимости **удаляет AAAA** для тех же имён.

1. В [панели NetAngels → API](https://panel.netangels.ru/account/api/) создайте **ключ API**.
2. Скопируйте `deploy/netangels.env.example` в **`.netangels.env`** в корне проекта (файл в `.gitignore`, не коммитить).
3. Заполните `NETANGELS_API_KEY`, при необходимости `NETANGELS_ZONE`, `TARGET_IPV4`, `DELETE_AAAA`.
4. Проверка без изменений: `npm run dns:netangels:dry`
5. Список записей API (отладка): `npm run dns:netangels:list`
6. Применить: `npm run dns:netangels`

Если структура имён в GeoDNS отличается (не `truewebwork.ru` / `www.truewebwork.ru`), сначала выполните `--list`, пришлите фрагмент JSON или поправьте логику в `deploy/netangels-dns-one-ip.mjs` под свои имена.

## Чеклист: один VPS — одно и то же на телефоне, планшете, ноутбуке и ПК

Сделайте по порядку (это и есть «всё до про Яндекс»):

1. **Aeza** — откройте карточку VPS и выпишите **публичный IPv4** (тот, по которому доступен сайт).
2. **NetAngels** (или где зона `truewebwork.ru`) — для **каждого** региона GeoDNS (**Россия**, **СНГ**, **Европа**, **мир / по умолчанию** и любые другие строки):
   - **A** для **`@`** → этот IPv4;
   - **A** для **`www`** → **тот же** IPv4 (или **CNAME** `www` → `@`, если так умеет панель).
3. **AAAA** — если у **`@`** (или `www`) есть IPv6, а на VPS **нет** настроенного IPv6, **удалите** эти AAAA (частая причина «на iPhone не открывается»).
4. **Сервер** — в `.env` на VPS: `NEXT_PUBLIC_SITE_URL=https://www.truewebwork.ru`, затем деплой (`npm run deploy:vps` или ваш скрипт).
5. **Проверка** — с **Wi‑Fi** и с **мобильного интернета** откройте `https://www.truewebwork.ru` (лучше инкогнито); с ПК — то же. Контент должен совпадать. `https://truewebwork.ru` при успешном DNS должен отдавать **301** на `www`.

**Клиент:** в проекте уже заложены адаптивная вёрстка, viewport, высота под iOS, карусель услуг со свайпом, редирект apex → `www` в middleware. После выравнивания DNS одна сборка одинаково открывается в Safari, Chrome и на больших экранах.

## VPS на Aeza и DNS: не путать панели

- **Aeza** — это **хостинг сервера** (Ubuntu, IP вида `138.124.xx.xx`). Здесь вы включаете машину, заходите по SSH, на сервере стоят nginx и приложение. **Записи DNS для домена `truewebwork.ru` в панели Aeza появляются только если сам домен куплен/делегирован на DNS Aeza.** Обычно домен сидит у регистратора (например NetAngels) — тогда **A/AAAA и GeoDNS правятся там**, а не в Aeza.
- **Один VPS (только Европа, без сервера в РФ):** во **всех** регионах GeoDNS (и для России тоже) для **`@` и `www` укажите **один и тот же** IPv4 вашего Aeza‑VPS. Иначе часть людей (мобильный интернет, РФ) резолвит домен на **старый или чужой IP**, а `www` — на рабочий: без `www` сайт «не открывается».
- **Цифры для сверки:** везде в DNS для `truewebwork.ru` в каждом регионе **`@` и `www` → тот же A, что вы видите в панели Aeza у сервера** (публичный IPv4).
- **AAAA:** если у **`@`** есть запись IPv6, а на VPS IPv6 не настраивали — **удалите AAAA** для `@` (и при необходимости для `www`), пока не настроите IPv6 на сервере.
- **Безопасность:** не отправляйте пароль root в чаты и мессенджеры. Смените пароль после утечки, включите вход по **SSH‑ключу**, при возможности отключите вход по паролю (`PasswordAuthentication no` в `sshd_config`).

## Этап 1–2 (уже в репозитории)

- Верификация: переменные `NEXT_PUBLIC_GOOGLE_SITE_VERIFICATION` и `NEXT_PUBLIC_YANDEX_VERIFICATION` в `.env` на **каждом** сервере (одинаковые токены).
- `robots.txt` открывает индексацию для всех и для Yandex, админку закрывает.
- Деплой:
  - `npm run deploy:vps` или `npm run deploy:vps:eu` — европейский VPS (`TRUWEB_VPS_HOST`).
  - `npm run deploy:vps:ru` — московский VPS (`TRUWEB_VPS_HOST_RU` в `.deploy.env`).

## Этап 3: NetAngels (ваша часть)

### Пошагово в панели NetAngels (DNS домена)

1. Войдите в [NetAngels](https://www.netangels.ru/) → раздел **домены** → зона **`truewebwork.ru`** → **DNS** / **редактор зоны** (названия могут чуть отличаться).
2. Найдите **геозависимые записи** (GeoDNS / «по регионам»). Откройте **каждый регион** (Россия, СНГ, «по умолчанию», мир и т.д.).
3. В **каждом** регионе проверьте:
   - **A** для **`@`** (корень, иногда «@» или пустое имя) → **IPv4 вашего VPS Aeza** (тот же, что в панели Aeza у сервера).
   - **A** для **`www`** → **тот же IPv4** (либо **CNAME** `www` → `@`, если провайдер это поддерживает и `@` уже указывает на нужный IP).
4. Запись **AAAA** для **`@`**: если есть и на VPS **нет** настроенного IPv6 — **удалите** AAAA для `@` (при проблемах на телефоне — и для `www` тоже).
5. Сохраните изменения, подождите **TTL** (или 15–30 минут) и с телефона в **мобильном интернете** откройте `https://truewebwork.ru/` — должен редиректить на `https://www.truewebwork.ru/`.

Если в NetAngels **нет** GeoDNS, а только обычные записи — добавьте/исправьте **две A** (или A + CNAME): и для `@`, и для `www` на **один** IP Aeza.

### Два VPS (когда появится сервер в России): Москва + Европа

Пока **только Aeza** — этот блок **не используйте**: держите везде один IP (см. раздел выше). Когда купите VPS в РФ и будете готовы — тогда РФ → IP Москвы, остальной мир → IP Европы, и **`npm run deploy:vps:ru`** + **`npm run deploy:vps:eu`** с одной и той же веткой кода.

1. Купить VPS в РФ, зафиксировать **публичный IPv4**.
2. Один раз выполнить `npm run deploy:vps:ru` с заполненным `.deploy.env` (см. `.deploy.env.example`).
3. В NetAngels для зоны `truewebwork.ru` включить **геозависимые записи** (если услуга называется «GeoDNS», «записи по регионам» или аналог — смотрите справку NetAngels).

   Типичная схема:

   - **Россия / СНГ (или «РФ»)** → A-запись на IP **Москвы**.
   - **По умолчанию (остальной мир)** → A-запись на IP **Австрии**.

   Важно: не делайте два разных «сайта» с разным контентом без продуманного `canonical` — у вас один код, два зеркала с одной сборкой.

4. TTL можно поставить 300–600 секунд на время проверки, потом поднять.

## SQLite и два сервера

Сейчас БД — **файл на каждом сервере**. Два деплоя = **две независимые базы** (заказы/админка на EU и RU разъедутся). Варианты на будущее:

- одна **общая** БД (PostgreSQL и т.д.) в одном месте;
- или основной приём заказов только на одном узле (сложнее в nginx).

Пока витрина и FAQ работают с минимальной записью — осознайте риск для корзины/ЛК.

## Про «Яндекс vs Google»

DNS смотрит **географию резолвера / пользователя**, а не «из какого поиска пришли». Для клиентов из России и Яндекс, и Google обычно откроют сайт с **российского IP** — это нормально и ожидаемо.

## Если у части людей сайт «не открывается» (Safari, Chrome, из Telegram)

Доступ **не зависит от кода сайта**, если проблема в сети до сервера. Проверьте по порядку:

1. **Оба зеркала живы** — с телефона откройте напрямую **IP Москвы** и **IP Европы** по `https://…` (или временно пропишите один IP в `/etc/hosts` / «запись в hosts» на ПК). Если один IP не отвечает или другой сертификат — друг с VPN/без может попадать именно на «плохой» узел из‑за GeoDNS или DNS VPN.
2. **Запись AAAA (IPv6)** — если в DNS есть AAAA, а на сервере IPv6 не настроен, часть клиентов (часто Apple) сначала ходит по IPv6 и видит таймаут. Либо настройте IPv6 на VPS, либо **уберите AAAA**, пока не готовы.
3. **Nginx** — в шаблоне репозитория для `location /` включено `proxy_buffering off` (стриминг Next.js). После правок: `sudo nginx -t && sudo systemctl reload nginx`.
4. **Один общий A** — для проверки можно временно оставить **одну** A‑запись на тот сервер, который точно работает; когда оба узла идентичны по TLS и приложению — вернуть GeoDNS.
5. **www и без www** — в DNS должны быть A (или CNAME) для **обоих** имён или редирект с `www` на основной домен; в `server_name` nginx перечислите оба, сертификат Let’s Encrypt — на оба. Иначе часть пользователей увидит ошибку или чужой виртуальный хост.
6. **«Application error» / пустая страница только на телефоне** — в коде избегаем `opacity: 0` в SSR у Framer Motion для основного контента; если после деплоя ошибка остаётся, смотрите консоль Safari (Web Inspector) и логи PM2 на сервере.

### «С www открывается, а без www (`truewebwork.ru`) — нет» (часто на телефоне, с ПК всё ок)

Это **не лечится только кодом**: редирект в Next.js срабатывает, только если запрос **уже пришёл на ваш VPS**. Если в DNS у **`@` другой IP**, чем у **`www`**, телефон открывает «пустой» или чужой сервер — редиректа не будет.

**Проверка за 30 секунд (на ПК):** в терминале выполните `nslookup truewebwork.ru` и `nslookup www.truewebwork.ru`. Строки `Address:` (IPv4) должны **совпадать**. Если нет — в панели NetAngels выровняйте записи.

**Если API NetAngels (`npm run dns:netangels:list`) показывает только две A на нужный IP, а симптом остаётся:** с внешних резолверов (8.8.8.8, 77.88.8.8) часто всё уже совпадает — тогда смотрите **кэш на устройстве**, **Wi‑Fi роутер (DNS)**, **Private DNS в Android**, другой **VPN**. Запустите из проекта: `powershell -NoProfile -File deploy/dns-smoke-check.ps1` — скрипт сравнит IP с нескольких DNS и проверит HTTPS.

**TLS:** если apex по HTTPS даёт ошибку сертификата, а `www` — нет, перевыпустите сертификат на **оба** имени (`certbot --nginx -d truewebwork.ru -d www.truewebwork.ru`) и проверьте `server_name` в nginx для порта 443.

**Nginx:** готовый пример с редиректом `http://truewebwork.ru` сразу на `https://www...` и с apex на 443 → www — файл `deploy/nginx-production-truewebwork.conf`. Частая ошибка: на80 для apex делают редирект только на `https://truewebwork.ru`, и у части клиентов второй шаг (TLS на apex) падает — в примере сразу канонический `https://www`.

**GeoDNS:** для **каждого региона** (РФ, «по умолчанию» и т.д.) набор A‑записей для **`@` и `www` должен быть одинаковым** (или `www` → CNAME на `@` после выравнивания `@`).

**Два VPS (Москва + Европа):** залейте сборку **на оба** (`npm run deploy:vps:eu` и `npm run deploy:vps:ru`). Иначе часть пользователей попадает на узел со старой конфигурацией.

Что сделать в панели NetAngels (или у регистратора):

1. Откройте DNS для `truewebwork.ru` и сравните **попарно**:
   - все **A** (и при GeoDNS — **для каждого региона**) для **`@`** и для **`www`**;
   - они должны вести на **те же IPv4**, что и рабочий `www`.
2. Если для `www` стоит **CNAME**, а для `@` — другой A или старый IP — выровняйте: либо одинаковые A на оба имени, либо `@` → нужный IP, `www` → CNAME на `@` (если провайдер так умеет).
3. Проверьте **AAAA (IPv6)** отдельно для `@` и `www`: если у `@` есть AAAA на «пустой» адрес, мобильные сети часто бьют в него первыми — временно **уберите AAAA** или настройте IPv6 на VPS.
4. После правок подождите TTL (или 10–30 минут) и с телефона в мобильном интернете откройте снова `https://truewebwork.ru/` (лучше в режиме инкогнито).

**Про Google:** в коде сайта канонический URL — `https://www.truewebwork.ru` (sitemap, meta, JSON‑LD). Запросы на `https://truewebwork.ru/…` приложение отвечает **301 на www** (middleware), если запрос дошёл до Next.js. Если DNS для `@` указывает не на тот же VPS, что и `www`, редирект не сработает — всё равно выровняйте A‑записи для `@` и `www`.

В Search Console после деплоя запросите переобход главной **с www** и при необходимости старого URL без www.

## Старый сниппет в Google

Поисковик кэширует выдачу; обновление занимает дни. Ускорить: **Google Search Console** → проверка URL `https://www.truewebwork.ru/` → «Запросить индексирование». Яндекс.Вебмастер — аналогично.
